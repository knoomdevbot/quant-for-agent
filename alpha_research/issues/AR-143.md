## Alpha Hypothesis
- **Claim:** Large forced-liquidation cascades can create temporary dislocations that mean-revert after liquidation intensity decelerates or price impact stalls.
- **Economic / behavioral intuition:** Forced deleveraging is non-discretionary flow. Once forced selling/buying exhausts, liquidity provision can earn reversal if entries are not too early and execution costs do not dominate.
- **Expected failure mode:** Signal is just generic short-horizon reversal/volatility spike; same-bar event timing leaks future information; taker fees, stressed spreads, and slippage erase edge.
- **Falsifier:** If timestamp-safe liquidation prints and tradable price/depth history are unavailable, mark blocked. If later event study fails after taker fees + high slippage or is dominated by simple reversal/vol-spike controls, reject.

## Implementation Plan
- **Candidate product pool:** BTC-PERP and ETH-PERP across venues with verified historical liquidation, trade, and depth coverage; SOL/top liquid perps only after coverage/depth gates pass.
- **Product-selection filters:** liquidation print availability by timestamp/side/size/exchange, minute or finer tradable prices, spread/depth proxy, venue reliability, sufficient event count.
- **Selected universe:** primary BTC/ETH perps until data gates justify expansion.
- **Universe-selection rationale / limitations:** crypto-native microstructure event; daily OHLCV is not enough for primary evidence.
- **Data source:** real liquidation/trade/order-book or minute-bar data from exchange APIs, CoinGlass, Kaiko, Amberdata, or equivalent. No fabricated proxies.
- **Signal features:** liquidation notional z-score/percentile, side, deceleration/exhaustion marker using only past/current data, price impact stall, liquidity/depth filter.
- **Model form:** feasibility-first event-study scout; later rule-based event reversal model.
- **Parameters / search space:** event percentile threshold, deceleration window, entry delay, holding horizon, stop/exit, slippage/taker-fee stress.
- **Position sizing:** small capped event positions, side-specific caps, liquidity-scaled notional.
- **Risk constraints:** exclude outage/illiquid intervals, stress slippage, cap clustered events, avoid same-bar lookahead.

## Evaluation Plan
- **Random-period protocol:** event-window and random-window tests across regimes, plus matched non-event windows.
- **Metrics:** event return distribution, Sharpe, hit rate, drawdown, turnover, adverse excursion, liquidity/depth stress, positive-window rate.
- **Costs/slippage assumptions:** mandatory maker/taker model with taker baseline; include maker-only diagnostic separately; 1x/2x/3x fee and stressed spread/depth assumptions.
- **Orthogonality checks vs alpha library:** compare to accepted liquidity-reversal and TSMOM streams when feasible; otherwise scout marks deferred.
- **Controls / ablations:** simple return reversal, vol spike, volume spike, funding/OI stress if available, random event labels, shifted liquidation timestamps, matched non-event windows, continuation leg.
- **Acceptance threshold:** feasibility passes only with timestamp-safe event data and enough events; later alpha pass requires positive median and non-materially-negative p25 after taker fees/high slippage, BTC-only and ETH-only survival, and superiority to simple reversal/vol-spike baselines.
