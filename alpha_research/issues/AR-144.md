## Alpha Hypothesis
- **Claim:** Large exchange inflows can forecast sell pressure, while large outflows can forecast reduced liquid float/accumulation, if on-chain exchange labels and publication lags are timestamp-safe.
- **Economic / behavioral intuition:** Moving coins to exchanges can increase immediately saleable supply; withdrawals can indicate accumulation/cold-storage demand. The mechanism is more direct than BTC/ETH price-only proxies.
- **Expected failure mode:** Provider labels are revised/non-PIT; flows are contemporaneous with price; signal is just momentum/reversal, volatility, or macro beta; small BTC/ETH universe lacks robust breadth.
- **Falsifier:** If timestamp-safe provider access/coverage cannot be confirmed, mark blocked. If later testing fails after conservative lag and maker/taker fees versus BTC/ETH momentum/reversal and stablecoin-liquidity controls, reject.

## Implementation Plan
- **Candidate product pool:** BTC and ETH spot/perp first; expand to SOL/BNB/XRP/large liquid majors only if exchange-label coverage and liquidity pass gates.
- **Product-selection filters:** provider metric availability, PIT/revision policy, conservative publication lag, coverage history, liquidity, fee schedule, venue reliability.
- **Selected universe:** BTC/ETH for feasibility; expansion only after data-quality gates.
- **Universe-selection rationale / limitations:** direct on-chain supply-pressure mechanism; not a cross-sectional altcoin data-mining exercise.
- **Data source:** real on-chain provider data only: Glassnode, CryptoQuant, Coin Metrics, Nansen, or equivalent. No OHLCV flow proxy and no CSV fixtures.
- **Signal features:** exchange netflow, inflow, outflow, exchange balance change, possibly whale/entity-filtered flows only if PIT-safe, conservative daily/weekly lags.
- **Model form:** feasibility-first scout; later daily/weekly rule-based long/flat/short or relative BTC/ETH allocation.
- **Parameters / search space:** flow z-score/lookback, lag, horizon, spot vs perp implementation, fee/funding assumptions.
- **Position sizing:** capped BTC/ETH exposure, volatility scaling, flat on missing/late metrics.
- **Risk constraints:** conservative data lag, no revised labels without disclosure, maker/taker fees, funding if perps, no shorting unless implementation supports it.

## Evaluation Plan
- **Random-period protocol:** daily/weekly random windows across market regimes after feasibility passes.
- **Metrics:** Sharpe, annualized return/vol, max drawdown, hit rate, turnover, positive-window rate, BTC/ETH leg diagnostics.
- **Costs/slippage assumptions:** mandatory maker/taker model for spot/perp execution; include funding if perps; 1x/2x/3x fee/slippage stress.
- **Orthogonality checks vs alpha library:** compare to crypto price TSMOM/reversal and accepted ETF risk/TSMOM if commensurable.
- **Controls / ablations:** BTC/ETH 1/7/30d momentum, reversal, realized volatility/drawdown, stablecoin liquidity, macro/risk beta, shifted-flow labels, inverted-flow direction.
- **Acceptance threshold:** feasibility passes only if provider metric availability and timestamp discipline are adequate; later alpha pass requires positive costed median random-window Sharpe, p25 not materially negative, and incremental value over controls.
